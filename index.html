INSPIRATION

A large part of my desire to become a game developer can be attributed to my favourite game: Ratchet & Clank: Going Commando (2003) for the Playstation 2. There are many reasons I love this game, but the reason I cite it as my inspiration to become a game developer can be attributed to a hidden location within the game known as the Insomniac Museum.

For anyone who hasn't played a Ratchet & Clank game, the Insomniac Museum is a location which appears throughout the series and features various interactive exhibits featuring aspects of the game's development alongside developer commentaries. Going Commando was the first game in the series to feature the location and includes content from both itself and the previous game, Ratchet & Clank (2002), with exhibits showcasing:

  • Developement resources - Notably a large testing area for the player's movement
  • Mechanics demonstrations - Particle effects, breakable meshes and gravity surfaces to name a few
  • Scrapped content - Models for weapons, enemies and objects which were ultimately unused
Alongside the exhibits, the accompanying developer commentaries explain, with an often humourous tone, what each exhibit represents and any challenges the item in question presented.

The museum left a lasting impression, showing the passion of the developers and showcasing work which may have otherwise gone unnoticed or unused, and provides a insight into the game development process at the time. It also showed me that the development process can be just as fun as the game itself ... and that programmers are often so hopped up on caffeine they may bounce off the walls.


For anyone who has played Ratchet & Clank: Going Commando, I would like to give a notable mention to the Planet Boldan "power line exploit" for allowing me to visit the museum at *reasonable* hours of the day

My responsibilities for this game were focused on development of the player abilities and player equipment. The game used a combination of Unreal's Gameplay Ability System (GAS) and a custom state machine approach to implement the character, which I used to implement both a death/respawn cycle and one of the player's primary movement abilities

Later in development, I was tasked with taking ownership of the weapon system and prototyping a new modular approach, which would allow the design team to quickly iterate and construct weapons. I collaborated with both programmers and designers to produce a prototype system, then presented the proposal in a feature review meeting

The modular weapon blueprint was comprised of UObject "modules", which fell into one of two categories:

  • Essential modules were for mechanics such as the firing mechanism (eg. automatic, semi-auto, manual, burst, etc.)
  • Optional modules were for mechanics such as FOV changes and player stat changes (eg. a movement speed decrease which could be used for heavy weapons)
These modules were then applied to sections of the blueprint which dictated when they would be active:
  • Shared modules were always active while the weapon was held
  • Hipfire modules were active while not aiming
  • Aiming modules were active while aiming
Hipfire and Aiming both had properties for Essential modules, which allowed the weapon to have different mechanics depending on aim (eg. changing to an underbarrel weapon while aiming), though Aiming would use the modules specified in Hipfire if left undefined

Azimov (named after famous science fiction writer Isaac Azimov) is a programming puzzle game which tasks the player with using a simple scripting language to program drones and complete various puzzles. The puzzles range from simple goals, such as helping a single drone reach a destination, to complex manipulation of multiple drones to accomplish tasks simultaneously

Azimov is set in a 3D grid-based environment and uses a lockstep timing system to ensure actions performed by all entities occur in "cycles", eg. a piston may operate over the space of 5 cycles: extending on the 1st cycle, remaining extended for 3 cycles, then retracting on the 5th cycle. The scriptable drones also conform to these cycles and their commands represent an action that can be performed in a single cycle, with optional parameters where appropriate. An example drone script could be:

  • MOVE 5 - Move forwards 5 spaces (each move will take 1 cycle)
  • TURN RIGHT - Turn 90° clockwise
  • GRAB - Grab the object in the space ahead of the drone
  • TURN LEFT - Turn 90° anticlockwise
  • MOVE 3 - Move forwards 3 spaces
  • DROP - Drop the held object in the space ahead of the drone
  • etc.
Once the drones have been scripted, the player can execute the program and watch the drone(s) perform the scripted actions in an attempt to complete the puzzle

My final project at university, this project is focused on a weapon which can be altered dynamically at runtime through the use of components and parameters

The modular weapon exists as a generic weapon with basic fire and reload functions, but the behaviour of the weapon is offloaded to it's components:

  • The Core defines the base weapon type and attributes and therefore only one can be equipped
  • Modifiers define various additional properties, such as Ricochet or Homing, and any number of these can be equipped simultaneously
Each component has various Parameters, which are UObject derived data classes that encapsulate various types (boolean, integer, float, custom, etc.). Parameters use delegates to allow the component to detect and react to changes, such as disabling other parameters or updating internal variables in response to a change. Through the in-game user interface, the player could both change the components on the weapon and view and modify the parameters of each component to rapidly iterate

The system also provides the ability to save to/load from a string, allowing users to share their creations, and was designed to be expanded to allow the users to save configurations in-game for later use

A group project from my final year, Stealing Stuff is an arcade-style game where players sneak around to steal everything that isn't nailed down and work together as a team to steal larger items. This project achieved 3rd place in both the Sumo Digital Award for Game Design and Rockstar Leeds Award for Best Team categories at the Game Republic Student Showcase 2020

I acted as the group's Lead Programmer for this project, which alongside my programming responsiblities also involved delegating tasks to other programmers on the team. I worked on multiple core systems of the game, including but not limited to:

  • Multiplayer Camera - The camera was designed to provide a 2.5D perspective and will adjust its distance to ensure that all active players are kept on screen
  • Character Controls - Third person character controls, with movement, sprinting, a dash and the pickup/drop abilities. The movement has 2 different types:
    • "Standard" movement is world relative, allowing the player to move based on cardinal directions
    • "Transport" movement is held object relative, allowing the player to push/pull the object with forward/backward input and rotate the object with sideways input
  • Object Interaction - Most objects in the game can be picked up, from small household items to large vehicles. Each object has a required "lift force", which is determined by it's weight, and this dictates how many players are required to lift an object and how fast they can move while doing so. While moving larger objects, the system determines a "pivot point" for the held object based on the respective grab locations of each player and feeds that information back to the players to drive their "Transport" movement mode
